# DtoOrm.Generator

A Roslyn incremental source generator that reads a `dtoorm.schema.json` file at build time and emits a `DtoOrm.Db.g.cs` file containing strongly-typed `Table` subclasses and `Column<T>` properties for every table and column in your schema.

## Installation

```bash
dotnet add package DtoOrm.Generator
```

Then declare the schema file as an `AdditionalFiles` item in your `.csproj`:

```xml
<ItemGroup>
  <AdditionalFiles Include="dtoorm.schema.json" />
</ItemGroup>
```

The generator looks for any file whose name ends with `dtoorm.schema.json` (case-insensitive). The file does not need to sit at the project root, but that is the conventional location.

---

## How it works

The generator runs during compilation as part of the Roslyn incremental pipeline. It reads the schema JSON from `AdditionalTexts`, deserializes it, and emits a single source file named `DtoOrm.Db.g.cs`. No external tooling, no T4 templates, no post-build steps.

Because it uses the incremental API, regeneration is gated on the schema file actually changing. Rebuilding without touching the schema file does not re-run the generator.

## Schema file format

The schema file is plain JSON. You typically generate it with the CLI tool, but you can write or edit it by hand.

```json
{
  "namespace": "MyApp.Data",
  "database": "mydb",
  "tables": [
    {
      "dbName": "users",
      "clrName": "Users",
      "alias": "u",
      "columns": [
        {
          "dbName": "id",
          "clrName": "Id",
          "clrType": "int",
          "ordinal": 1
        },
        {
          "dbName": "email",
          "clrName": "Email",
          "clrType": "string",
          "ordinal": 2
        },
        {
          "dbName": "is_active",
          "clrName": "IsActive",
          "clrType": "bool",
          "ordinal": 3
        }
      ]
    }
  ]
}
```

### Top-level fields

| Field | Type | Required | Description |
|---|---|---|---|
| `namespace` | string | no | C# namespace for the generated `Db` class. Defaults to `DtoOrm.Generated`. |
| `database` | string | no | Database name. Informational only; not used in code generation. |
| `tables` | array | yes | List of table definitions. |

### Table fields

| Field | Type | Required | Description |
|---|---|---|---|
| `dbName` | string | yes | Name of the table in the database. |
| `clrName` | string | yes | Name used for the generated class and the static property on `Db.Tables`. |
| `alias` | string | no | SQL alias used in queries. If omitted, the generator derives one from the first three alphanumeric characters of `dbName`. |
| `columns` | array | yes | List of column definitions. |

### Column fields

| Field | Type | Required | Description |
|---|---|---|---|
| `dbName` | string | yes | Name of the column in the database. |
| `clrName` | string | yes | Property name in the generated class and the alias used when selecting. |
| `clrType` | string | yes | Fully-qualified or short CLR type name (e.g. `int`, `string`, `System.DateTime`). |
| `ordinal` | int | no | Column position; used to order the column properties in the generated file. |

The `clrType` string is emitted verbatim into the generated `Column<T>` declaration, so you can use any type your project can resolve, including nullable types (`int?`, `string?`) and custom structs.

## Generated code structure

Given the schema above with `namespace: "MyApp.Data"`, the generator emits:

```csharp
// <auto-generated />
#nullable enable

namespace MyApp.Data;

public static partial class Db
{
    public static class Tables
    {
        public static readonly UsersTable Users = new();
    }

    public sealed class UsersTable : global::DtoOrm.Core.Table
    {
        internal UsersTable() : base(@"users", @"Users", @"u")
        {
            Id       = new global::DtoOrm.Core.Column<int>(this, @"id", @"Id");
            Email    = new global::DtoOrm.Core.Column<string>(this, @"email", @"Email");
            IsActive = new global::DtoOrm.Core.Column<bool>(this, @"is_active", @"IsActive");
        }

        public global::DtoOrm.Core.Column<int>    Id       { get; }
        public global::DtoOrm.Core.Column<string> Email    { get; }
        public global::DtoOrm.Core.Column<bool>   IsActive { get; }
    }
}
```

Key points:

- `Db` is declared `partial`, so you can add hand-written members in the same namespace alongside the generated file.
- Each table is a nested class within `Db`, not a top-level type.
- The singleton instances live on `Db.Tables`.
- Column constructors are called inside the table constructor so that `this` (the `Table` reference) is available.

## Using the generated types

```csharp
using MyApp.Data;

// Grab the table reference once; it's a singleton
var users = Db.Tables.Users;

var activeUsers = await session
    .From(users)
    .Select(users.Id, users.Email)
    .Where(users.IsActive.Eq(true))
    .ToListAsync<UserDto>();
```

Because each `Column<T>` property is typed, the condition methods are typed too. `users.IsActive.Eq("wrong")` is a compile-time error.

## Refreshing the schema

When you change the database schema, run the CLI again to overwrite `dtoorm.schema.json`:

```bash
dtoorm schema \
  --connection "Server=localhost;Port=3306;Database=mydb;User ID=root;Password=secret;" \
  --output dtoorm.schema.json \
  --namespace MyApp.Data
```

Then rebuild. The generator picks up the new file automatically.

Commit `dtoorm.schema.json` to source control. It is the contract between your database and your C# code, and its history is useful for reviewing schema changes alongside code changes.

## Extending the generator

`DtoOrmSourceGenerator` is declared `sealed`. The extension point is the schema file itself: it is plain JSON, so any tooling can produce or transform it.

If you need a different code shape — for example, generating repository classes alongside the table definitions — add a second source generator in your own project that reads the same `dtoorm.schema.json` file from `AdditionalTexts` and emits whatever additional source you need. The two generators operate independently and their outputs are merged by the compiler.

The `Db` class is `partial`, which is specifically intended to let you add extension methods, factory helpers, or other hand-written members without modifying generated files.
